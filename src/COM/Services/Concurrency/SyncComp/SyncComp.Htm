Object Synchronization Time Comparison

COM+ Synchronization Overview

COM+ provides synchronization from component to component when called from a multi-threaded client without the complexity of writing a free-threaded object. By setting the synchronization attribute for a component you can let the COM+ Services handle the component synchronization. You can read more about COM+ synchronization in the Platform SDK under COM+ (Component Services), Concurrency.

Sample Overview

The SyncComp sample shows the time performance of the COM+ Synchronization service in relation to the other models, Single Threaded Apartment and Free Threaded Apartment. This sample comprises of a host which creates either a Single Threaded Apartment object, a Free Threaded Apartment object or a COM+ Synchronized object and then creates multiple threads to call the created object multiple times. The elapsed time required to call each type of object is recorded and averaged. This sample shows that a COM+ Synchronized object is slightly slower than a Free Threaded object but faster than an Apartment Threaded object. This shows that you get almost the same speed benefit of a free-threaded object while maintaining the simplicity of writing single threaded objects.

Project Files

Sub-directory Description
SyncComp Main dialog which creates the desired COM objects and the threads to exercise those objects.
SyncComp\STAObject COM marked as Apartment Threaded. No special synchronization code. Standard COM Apartment synchronization keeps things Synchronized.
SyncComp\FreeObject COM marked as Free Threaded. A Critical Section is used to maintain synchronization.
SyncComp\ComSyncObject COM+ object marked as Free Threaded but has no special synchronization code. COM+ synchronization provides the synchronization.

Build Notes

  1. Build the SyncComp client and the three objects by opening the SyncComp.DSW workspace from Visual C++, set "SyncComp Classes" as the active project, then build by pressing the "F7" key. You can also use the Microsoft NMAKE-compatible makefile (MAKEFILE).

  2. Create a new empty COM+ Server Application called STA Object.
  3. Add the STAObject.DLL as a new component to the application.
  4. On the Activation tab of the properties for the STAObject component un-check the Enable Just In Time Activation option.
  5. On the Concurency tab set the Synchronization support to Not Supported.
  6. All other property defaults are acceptable

  7. Create a new empty COM+ Server Application called Free Object.
  8. Add the FreeObject.DLL as a new component to the application.
  9. On the Activation tab of the properties for the FreeObject component un-check the Enable Just In Time Activation option.
  10. On the Concurency tab set the Synchronization support to Not Supported.
  11. All other property defaults are acceptable

  12. Create a new empty COM+ Server Application called ComSync Object.
  13. Add the ComSyncObject.DLL as a new component to the application.
  14. On the Activation tab of the properties for the ComSyncObject component un-check the Enable Just In Time Activation option.
  15. On the Concurency tab set the Synchronization support to Required.
  16. All other property defaults are acceptable

Run Notes

To run this sample

  • Start the client program (SyncComp.EXE).
  • Change the number of threads and/or the number of iterations to desired levels.
  • Make sure the radio button next to STA is selected.
  • Press the start button to run the tests. The results will be recorded in the text boxes to the right of the radio button.
  • Select the radio button next to Free-Threaded and press the start button. Let the tests finish.
  • Select the radio button next to the COM+ Sync and press the start button. Let the tests finish.
Note that the COM+ times are only slightly slower then the free-threaded mode but faster than the STA mode. You might need to adjust the number of threads or the iterations (or both) to see meaningfull results.

Remarks

Another benefit of COM+ synchronization is that it can nest synchronization domains. To explain this concept imagine a series of objects (O1, O2, O3) that can be called directly by the client or can be called by each other. Since the objects can be called by the client each object must be able to execute in a synchronized domain. In a free-threaded model each object would create it's own synchronization. So, in the case where the client calls O1, O1 calls O2, and then O2 calls O3, there will be three synchronization objects created (one for each object being called). However, if these objects where marked as COM+ Synchronized objects (written without the synchronization code) and the same example is run: client calls O1, O1 calls O2, and then O2 calls O3. Only one synchronization object will be created. This is because when O1 calls O2 COM+ recognizes that the call is already in a synchronization domain. So, COM+ doesn't need to create another synchronization object. The same is true when O2 calls O3.
